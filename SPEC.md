# SPEC.md — Backend технического задание

Self-hosted приложение управления инвестиционным портфелем.
Этот документ — единственный источник истины. При конфликте требований —
спросить, не решать молча.

Объём первой итерации: **MVP-ядро** — транзакции, позиции, портфели, цены.
Evidence-based аналитика (MPT, DCF, Gordon, Black–Scholes) в MVP **не реализуется**:
вместо неё — типизированные интерфейсы-заглушки с TODO, чтобы добавить отдельной фазой.

## 2. Доменная модель (реализовать точно)

### Transaction — фундаментальная единица
Три типа:

- **input** — поступление актива в систему (депозит, получение крипты).
  Создаёт/пополняет позицию из внешнего источника.
- **transfer** — обмен одного актива на другой (покупка/конвертация).
  **Именно transfer создаёт позиции**: списывает исходный актив,
  создаёт/увеличивает позицию по целевому. Хранит исходный актив/валюту.
- **output** — вывод актива из системы (продажа, перевод наружу).
  Уменьшает/закрывает позицию.

### Position — создаётся автоматически из transfer
Хранит: asset, quantity, entry_price, currency (из которой делался transfer),
timestamp создания. Для позиции трекается изменение цены актива и количества
относительно базовой валюты с момента создания.
Нужен расчёт **PnL и доходности** на уровне отдельной позиции (endpoint + сервис).

### Portfolio — пользовательская группировка позиций
Произвольная (по стратегиям, классам активов). Агрегирует связанные транзакции
и активы. Endpoint дашборда портфеля: суммарная стоимость, PnL по позициям.
Сложная аналитика — **заглушки-интерфейсы с TODO**, не реализовывать в MVP.

### Asset — связан с источником цены
Поля: symbol, name, asset_class (`stock` / `stock_ru` / `crypto` / `custom`),
pricing_provider.

### User + auth
Простая JWT-аутентификация: регистрация, логин, защищённые endpoints.
Пароли — argon2 или bcrypt. Без OAuth/SSO.

## 3. Pricing providers

Абстрактный интерфейс `PriceProvider` с методами получения текущей и
исторической цены. Реализации:

- **MoexProvider** — публичный API Московской биржи (без ключа), для `stock_ru`.
- **CoingeckoProvider** — публичный Coingecko API, для `crypto`.
- **YahooProvider** — зарубежные инструменты, для `stock`.
- **CustomProvider** — пользовательский источник цены по предоставленной
  модели интеграции (для `custom`-активов с динамической ценой).
- **ManualProvider** — цена задаётся вручную, без внешнего API
  (недвижимость, доли в бизнесе, неликвид).

Все внешние вызовы: таймаут, graceful degradation, не ронять сервис при
недоступности источника.

---

## 4. Фоновые задачи (Celery + Redis)

- Периодическое обновление цен активов.
- Снэпшоты портфелей: стоимость портфеля во времени → отдельная timeseries-
  таблица (основа для будущих графиков на фронте).
- Celery worker + Celery beat — отдельные сервисы в docker compose.

---

## 5. Архитектурные требования

- Структура: `routers / models / schemas / services / providers / tasks`.
- Бизнес-логика — в `services`, не в роутерах.
- Конфиг через переменные окружения, обязателен `.env.example`.
- Единый формат ошибок API.
- OpenAPI-схема доступна (FastAPI генерит автоматически — проверить, что
  все endpoints типизированы и видны в `/docs`).

## 7. Acceptance (проверяется в КАЖДОЙ применимой итерации)

Итерация не закрыта, пока не выполнено всё применимое:

1. **Docker compose реально поднимается**: `docker compose up` стартует все
   сервисы без падений; backend отвечает на healthcheck.
2. **Миграции Aerich применяются**: `aerich upgrade` проходит на чистой БД
   без ошибок; повторный прогон идемпотентен.
3. **Тесты зелёные**: `pytest` проходит полностью, ключевые сценарии:
   - transfer-транзакция создаёт позицию;
   - расчёт PnL по позиции корректен (с фиксированными ценами);
   - output закрывает/уменьшает позицию;
   - auth-флоу (регистрация → логин → защищённый endpoint).
4. **OpenAPI доступен**: `/docs` и `/openapi.json` отдаются, все endpoints
   типизированы, нет «голых» dict без схем.

В конце итерации 6 дать единый чек-лист команд, которыми пользователь
локально проверит пункты 1–4 и end-to-end сценарий через curl/httpie.

## 8. Чего НЕ делать в MVP

- Реализовывать математику MPT / DCF / Gordon / Black–Scholes
  (только типизированные заглушки с TODO).
- OAuth/SSO, внешние identity-провайдеры.
- Фронтенд (отдельный проект).
- Платёжную/биллинговую логику (продукт бесплатный, self-hosted).
